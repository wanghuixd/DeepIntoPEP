# PEP 426：Python 软件包元数据 2.0

PEP 426 是 Python 增强提案，旨在定义 Python 软件分发的**元数据规范 2.0**。作者为 Nick Coghlan、Daniel Holth、Donald Stufft，于 2013 年提出。该 PEP 现已被**撤回**（Withdrawn），但其设计思路对后续规范（如 PEP 566）产生了重要影响。

## 背景与动机

在 PEP 426 之前，Python 包元数据沿用自 PEP 345 的 **Key-Value 文本格式**（类似 RFC 2822 邮件头），例如：

```
Name: my-package
Version: 1.0.0
Summary: A short description
```

这种格式存在若干局限：

- **解析复杂**：需要专门解析器处理多行值、编码等；
- **扩展困难**：新增字段缺乏统一机制；
- **工具链割裂**：不同工具（pip、setuptools、PyPI）对元数据的处理方式不一。

PEP 426 提出将元数据迁移到 **JSON 兼容的内存表示**，以简化解析、便于扩展，并与现代工具链更好集成。

## 核心提案

### 1. JSON 化元数据

将传统的 Key-Value 格式改为 JSON 结构，例如：

```json
{
  "name": "my-package",
  "version": "1.0.0",
  "summary": "A short description",
  "description": "Extended description in Markdown or reStructuredText",
  "keywords": ["tag1", "tag2"],
  "license": "MIT",
  "requires_dist": ["requests>=2.0"],
  "requires_python": ">=3.8"
}
```

### 2. 主要元数据字段

| 字段 | 说明 |
|------|------|
| `name` | 包名，主要标识符 |
| `version` | 版本号（遵循 PEP 440） |
| `summary` | 简短描述 |
| `description` | 详细描述（支持 Markdown/reStructuredText） |
| `keywords` | 关键词列表 |
| `license` | 许可证 |
| `requires_dist` | 依赖列表（遵循 PEP 508） |
| `requires_python` | Python 版本要求 |
| `author` / `maintainer` | 作者/维护者信息 |
| `project_url` | 项目链接（如 Home-Page、Documentation） |

### 3. 扩展机制

PEP 426 引入**正式扩展机制**，允许在不修改核心规范的前提下新增字段。扩展字段以特定前缀命名，便于工具识别与忽略未知字段。

## 与相关 PEP 的关系

PEP 426 建立在以下规范之上：

- **PEP 425**：二进制兼容性标签（wheel 平台标签）
- **PEP 427**：Wheel 分发格式
- **PEP 440**：版本标识与依赖规范
- **PEP 503**：PyPI 简单 API
- **PEP 508**：依赖说明符语法
- **PEP 518**：构建系统依赖（pyproject.toml）

## 为何被撤回

PEP 426 于 2013 年 12 月起被**推迟**，至 2017 年 3 月正式**撤回**。主要原因包括：

1. **迁移成本过高**：从 Key-Value 全面切换到 JSON 涉及 pip、setuptools、PyPI、wheel 等整个生态，缺乏可行的渐进迁移路径。
2. **社区共识**：包装社区倾向于**增量改进**而非推倒重来。
3. **替代方案**：**PEP 566** 采用更温和的方式——保留原有 Key-Value 格式，同时定义将其转换为嵌套 JSON 结构的标准机制，既兼容现有工具，又为未来扩展留出空间。

## 历史意义与启示

尽管 PEP 426 未获采纳，其设计思想仍有参考价值：

- **JSON 化**：现代工具（如 `importlib.metadata`）已支持将元数据解析为结构化数据；
- **扩展机制**：PEP 566 及后续规范吸收了“可扩展元数据”的思路；
- **规范演进**：大型规范变更需要兼顾**向后兼容**与**渐进迁移**，单次大改往往难以落地。

> 当前核心元数据规范（如 packaging.python.org 上的 Core Metadata）仍以邮件头式 Key-Value 为主，版本号已演进至 2.x。若需了解最新字段与格式，可查阅 [Core Metadata 规范](https://packaging.python.org/en/latest/specifications/core-metadata/)。
